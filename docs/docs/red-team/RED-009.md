# RED-009: Prototype Parity & Fiscal Intelligence (Red Team Analysis)

## 1. Attack Surface: Advanced Logic Integrity
This strategy introduces **Inverse Calculation** and **Fiscal Modeling**, both of which significantly increase the risk of precision errors and logic loops.

## 2. Theoretical Attack Vectors
| Vector | Description | Mitigation Strategy |
| :--- | :--- | :--- |
| **Inverse Divergence** | A target pension that is mathematically impossible (e.g., $5,000,000 MXN) causes the solver to loop infinitely. | Implement a maximum iteration cap (100) and "Solution Unattainable" fallback logic. |
| **Tax Bracket Collision** | Rounding errors at the 15 UMA boundary for ISR exemption lead to incorrect net pension display. | Use precise UMA constants and double-verify against the 2026 Tax Table. |
| **Document Injection** | Malicious profile data (XSS) being injected into the "Escritos" (Legal Templates). | String sanitization and PDF-safe template rendering. |
| **Inflation Loop** | Compounding inflation costs incorrectly (e.g., adding 4% to the total instead of each year). | MATH-009 formalizes the compounding $(1+r)^t$ formula. |

## 3. High-Risk Scenarios
* **Target Pension < Base Pension**: User sets a target lower than their current projection. The "Inverse Builder" might try to reduce salary or years (illegal per IMSS rules).
* **Mitigation**: Solver must be constrained to only monotone increases from the baseline.

## 4. Continuity Risk
* **Risk**: The legacy prototype uses hardcoded 2026 values.
* **Mitigation**: Move all prototype "Constants" to the `legal-anchors.json` system for multi-decade durability (Protocol 23).
