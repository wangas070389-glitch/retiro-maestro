# RED-010: Red Team Analysis - Server-Side Migration

## 1. Attack Vectors & Risks

### A. The "Vercel Tax" (Infrastructure Lock-in)
*   **Attack**: Relying heavily on Next.js/Vercel primitives (Edge Config, KV, Server Actions) creates a dependency that is hard to port to a standard Docker/VPS setup.
*   **Impact**: High hosting costs at scale; loss of "Sovereign" portability.
*   **Mitigation**: Use standard Docker-compatible builds. Avoid Vercel-specific proprietary hooks (use standard Postgres, standard Redis if needed).

### B. The "Latency Illusion" (UX Degradation)
*   **Attack**: Moving the `PensionEngine` (currently instant client-side) to a Server Action introduces network latency (50ms - 200ms).
*   **Impact**: "La Escalera" and sliders feel sluggish/janky. The "Interactive" feel is lost.
*   **Mitigation**: **Hybrid Compute**. Keep the "Estimation Engine" on the Client for instant feedback, but use the Server Engine for "Final Verification" and persistence.

### C. Complexity Explosion (State Sync)
*   **Attack**: Synchronizing Client State (Zustand) with Server State (DB) leads to "Split Brain" bugs where the UI shows one value and the DB has another.
*   **Impact**: User trust erodes immediately if "Save" doesn't match "View".
*   **Mitigation**: **Optimistic UI** patterns + "Server Authority" model. The Server Response always overwrites Client State on save.

## 2. Pre-Mortem (Failure Scenarios)

### Scenario 1: "The Database is Down"
*   **Event**: Postgres connection fails.
*   **Result**: App becomes unusable even for simple calculations.
*   **Fix**: Implement **Offline Mode**. The app should fallback to Client-Side logic if the server is unreachable, syncing later.

### Scenario 2: "Session Hijack"
*   **Event**: User A guesses User B's Simulation ID.
*   **Result**: Data leak of sensitive financial info.
*   **Fix**: Row Level Security (RLS) in Postgres + UUIDv4 for IDs + strict Session Ownership checks.

## 3. Vulnerability Score
*   **Severity**: High (Architectural Pivot)
*   **Likelihood**: Medium
*   **Mitigation Status**: **Plan Required** for Hybrid Compute (Client for speed, Server for truth).

---
## 🛡️ Counter-Measures Adopted
1.  **Hybrid Engine Strategy**: We will NOT simply move everything to the server. We will duplicate the lightweight calculation logic on the client for UX, validation on server.
2.  **Docker-First**: All server code must run in a standard Node container, no serverless-only dependencies.
3.  **RLS Policies**: Database security is paramount.
