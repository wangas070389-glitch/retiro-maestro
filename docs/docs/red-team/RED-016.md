# RED-016: Threat Model for Client-Side PDF Parsing

## 1. Executive Summary
By shifting PDF extraction from the server to the client (edge) to protect Data Sovereignty, we alter the attack surface. This document analyzes the vulnerabilities introduced by `pdfjs-dist` in the browser and the mitigations applied via `SentinelAuditor`.

## 2. Identified Threats

### T1: Malicious PDF Payloads (Client-Side DoS / XSS)
- **Vector**: Adversary uploads a specially crafted PDF designed to exploit vulnerabilities in `pdfjs-dist` or cause an infinite loop in the browser worker (Resource Exhaustion).
- **Mitigation**: 
  - Restrict file size to <5MB in the dropzone.
  - Execute `pdfjs-dist` in a WebWorker to prevent locking the main UI thread.
  - Rely on CSP (Content Security Policy) to prevent execution of embedded scripts.

### T2: Heuristic Evasion (Data Poisoning)
- **Vector**: Adversary modifies the plaintext layer of the IMSS PDF to bypass `SentinelAuditor` checks (e.g., faking 3000 weeks while keeping age logic consistent).
- **Mitigation**: 
  - Client-side extraction is treated as **P0 (Unverified Truth)**. 
  - Financial projections are generated, but the *Official Dossier Stamp* requires a backend "Forensic Handshake" (Future feature) to verify the digital signature of the true IMSS PDF against governmental servers.

### T3: Dependency Hijacking
- **Vector**: `pdfjs-dist` is compromised via supply chain attack.
- **Mitigation**: 
  - Lock `pdfjs-dist` version in `package.json`.
  - Use Subresource Integrity (SRI) if loading the worker from a CDN, but preferably bundle the worker locally.

## 3. Conclusion
Client-side parsing significantly increases user privacy (PII never leaves the browser during exploration). The residual risk of DoS attacks against the local client is acceptable, given that it does not impact server health.
