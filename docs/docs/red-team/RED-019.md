# RED-019: Admin Control Panel Attack Vectors & Lockout Mitigation

## Overview
The introduction of the Admin User Control Panel (`STRAT-030`) creates a highly centralized command center capabilities. While it secures the application against unauthorized public access, it simultaneously introduces a single point of failure and severe lateral movement risks if an Admin account is ever compromised.

## Identified Threat Vectors

### 1. The "Self-Guillotine" (Accidental Admin Lockout)
**Vulnerability:** An administrator could inadvertently (or maliciously via CSRF if not protected) toggle their own `isBlocked` flag to `true`, or downgrade their `role` to `USER`. If no other admins exist, the system is permanently soft-bricked until a DB-level manual intervention occurs.
**Mitigation:** 
- The Server Actions (`admin-actions.ts`) MUST include a strict guard checking `if (targetUserId === session.user.id) throw new Error("Acción prohibida sobre cuenta propia");`. Admins cannot modify their own clearance or block status via the UI.

### 2. Privilege Escalation via Password Reset
**Vulnerability:** A rogue or compromised admin could reset the password of a Sovereign-tier user to gain access to their highly sensitive forensic dossiers, or reset the password of *another* Admin to execute a complete system takeover.
**Mitigation:**
- Implement a rigid hierarchy check: `ADMIN` roles cannot act on other `ADMIN` accounts unless they have a `SUPER_ADMIN` flag (if introduced in the future), or conversely, prevent admins from resetting the passwords of other admins entirely. 
- All password resets MUST generate a systemic log (even if just `console.warn` for now) citing the Admin ID and the Target ID.

### 3. API Endpoint Bypasses
**Vulnerability:** The `/admin/users` UI is protected, but if the underlying Server Actions do not independently verify the JWT role, an attacker with a standard `USER` token could forge requests to `toggleUserApprovalAction(adminId, false)`.
**Mitigation:**
- Double verification requirement. The layout middleware protects the UI, but every exported Server Action must initiate with `const session = await auth(); if (session?.user?.role !== 'ADMIN') throw new Error("Unauthorized");`.

## Protocol Directives
The implementation of the Super Tablero must adhere to the **Rule of Redundancy**. Access control is evaluated twice: once at the edge (UI routing) and once at the core (Server Action execution).
