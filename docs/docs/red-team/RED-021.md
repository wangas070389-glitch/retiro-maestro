# Sovereign Advisor Access Vectors (RED-021)

## Attack Surface Analysis
The B2B2C pivot introduces the `ADVISOR` role, a semi-privileged tier that acts as an intermediary. Unlike standard users, an Advisor requires read/write access to *other* users' data (their clients' Dossiers). This drastically expands the authorization threat model.

### 1. Lateral Access Escalation (Snooping)
*   **Vector:** An authenticated Advisor attempts to modify URL parameters or API payloads (IDOR) to fetch the simulation `strategyId` of a user who is **not** assigned to them.
*   **Mitigation:** All queries executed under the context of an `ADVISOR` role must include a mandatory `WHERE` clause verifying the relationship. For example:
    `db.user.findFirst({ where: { id: targetUserId, advisorId: session.user.id } })`. If `advisorId` does not strictly match the session's ID, the request must fail with a 403 Forbidden.

### 2. Upward Privilege Escalation (Self-Promotion)
*   **Vector:** A standard `USER` manipulates their local JWT or intercepts a profile update request to inject `role: 'ADVISOR'` or `advisorId: <admin_id>`, attempting to bypass payment walls or gain CRM access.
*   **Mitigation:** Only the `ADMIN` role via the Super Tablero (`src/actions/admin-actions.ts`) possesses the authority to mutate the `role` and `advisorId` fields. The standard `updateProfileInfoAction` must rigidly destructure and ignore any unauthorized fields.

### 3. Impersonation Abuse (Audit Trail Corruption)
*   **Vector:** An Advisor modifies a client's Dossier maliciously or erroneously. Since the Advisor is acting "on behalf" of the client, the system logs the modification as if the client performed it.
*   **Mitigation:** Any simulation save or dossier mutation executed by an Advisor on a client's behalf must be marked in the payload or via an audit log. The system should ideally record the `authorId` (the Advisor who clicked save) separately from the `ownerId` (the Client whose dossier holds the save).

### 4. Assignment Hijacking
*   **Vector:** An attacker attempts to set their own `advisorId` to a high-profile Advisor to gain free consultations, or an Advisor attempts to forcibly assign unconsenting users to their portfolio.
*   **Mitigation:** The assignment of an `advisorId` to a `User` is a restricted operation. It must either be performed by an `ADMIN` or via a secure cryptographic invite link generated explicitly by the Advisor and consumed by the User.
