# ADR-019: Sovereign Identity & RBAC Architecture

## Status
ACCEPTED

## Context
The platform has evolved from a direct-to-consumer (B2C) calculator to a **B2B2C SaaS ecosystem** (STRAT-032). This demands a layered identity architecture that supports three distinct personas — end-users, financial advisors, and administrators — while preserving Sovereign Privacy (Protocol 0) and cryptographic data isolation.

The original PROPOSED version of this ADR assumed a simple multi-tenant model. The B2B2C pivot (POST-010) revealed that the **Advisor** is the primary power user: a semi-privileged actor who manages a portfolio of client dossiers. This fundamentally changes the identity threat model.

## Decision
Adopt **NextAuth.js v5 (Auth.js)** with a **Polymorphic User Model** and **Role-Based Access Control (RBAC)**.

### 1. Authentication Foundation
- **Framework**: NextAuth.js v5 integrated with the Next.js App Router.
- **Providers**: Email/Password (bcrypt-hashed) via Credentials provider. OAuth (Google/Microsoft) as future extension.
- **Session**: JWT-based sessions. Custom `role` and `tier` fields are explicitly mapped into the JWT callback so the frontend can route users correctly without additional DB calls.

### 2. Polymorphic User Model (ADR-034)
A single `User` table supports three roles via a `role` scalar field and a self-referential `advisorId` foreign key:
- **`USER`**: Standard end-user. Has access to their own simulations and dossiers only.
- **`ADVISOR`**: Financial consultant. Has read/write access to the dossiers of assigned clients (`WHERE advisorId = session.id`). Upgraded from USER by an ADMIN via the Super Tablero.
- **`ADMIN`**: System operator. Full access to the Admin Control Panel (`/admin`). Manages user lifecycle (approve, block, reset password, change role/tier).

### 3. Gated Sanctuary Registration (STRAT-030)
New registrations are placed in a **pending** state (`isApproved: false`). Access to the simulation engine is gated until an `ADMIN` manually approves the account. This prevents algorithmic scraping and protects the actuarial IP.

### 4. Data Isolation
- Every `Simulation`, `Strategy`, and `ForensicDossier` record is keyed by `userId` at the Prisma schema level.
- Advisor queries must include `WHERE advisorId = session.user.id` — server actions enforce this, not the UI.
- Row Level Security (RLS) is recommended as a defense-in-depth layer at the Postgres level.

### 5. Sovereign Handshake (Protocol 23)
Users may "Disconnect" their cloud account and export all forensic data as portable JSON bundles, reverting to local-first "Sovereign Mode" without requiring a cloud session.

## Consequences
- **Positive**: Unified authentication across all three personas; no separate auth stacks for advisors vs users.
- **Positive**: Fluid upgrade path — a `USER` becomes an `ADVISOR` by a single `role` string mutation.
- **Risk**: With `ADVISOR` access to client data, IDOR attacks become the primary threat vector. Mitigated by strict `WHERE advisorId` server-side guards (RED-021).
- **Risk**: Admin self-lockout. Mitigated by server action guards: `if (target.id === session.user.id) throw` (RED-019).

## Superseded Concepts
The original "Magic Links" and simple tenant isolation model has been replaced by the full RBAC + Polymorphic User architecture defined above.
