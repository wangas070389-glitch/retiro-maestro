# ADR-029: Inyección Soberana del Oráculo vía RBAC

## Contexto
El sistema de proyecciones requiere valores precisos como la UMA (Unidad de Medida y Actualización) para funcionar. Originalmente, contemplamos conectar el *Economic Oracle* a la API del INEGI (ADR-022). Sin embargo, esto introduce una dependencia a una red externa de gobierno, violando el principio de **Soberanía Computacional**: si la API del INEGI cae, nuestro SaaS quedaría paralizado. 

## Decisión Arquitectónica
Rechazamos la conexión síncrona/dinámica a la API del INEGI. En su lugar, hemos decidido crear una interfaz de inyección manual hiper-segura ("Settings / Oracle Console") exclusiva para Administradores, amparada por un sistema de Control de Acceso Basado en Roles (RBAC).

1. **Expansión del Schema (Persistencia de Privilegio):**
   - Inyección de una columna `role` (`String @default("USER")`) en el prisma model de `User`.
   - Sólo los perfiles de base de datos definidos nativamente como `"ADMIN"` podrán interactuar o incluso conocer la existencia de la ruta de configuración global.

2. **Cámara de Vacío y Server Actions Restringidos (`/settings`):**
   - Invocaremos un middleware a nivel de Componente de Servidor reactivo (`app/(main)/settings/page.tsx`): `if (user.role !== 'ADMIN') notFound()`. Esta estrategia garantiza que atacantes malintencionados ni siquiera reciban un mensaje `401 Unauthorized` que podría incitar fuzzing, sino un velo ciego de inexistencia (`404 Not Found`).
   - El *Server Action* (`updateManualAnchorAction`) que ejecuta la inyección de SQL hacia `EconomicAnchor` llevará el mismo blindaje de validación asíncrono.

3. **Inversión de Prioridad (Source of Truth Override):**
   - El método original `OracleService.fetchLatestAnchors()` de la capa de Lógica de Negocio dejará de retornar un mock estático 2024 local.
   - Leerá agresivamente (vía Prisma) la base de datos SQL primero.
   - Si (y solo si) no existe registro inyectado por el Admin en la base de datos, entonces caerá a nuestra copia local codificada en la construcción (`legal-anchors.json`) previniendo un colapso del simulador si la base de datos se borrase.

## Consecuencias Positivas
- Escalabilidad Infinita de Soberanía. El sistema no hace llamadas HTTP dependientes fuera del contenedor web.
- El administrador maestro tiene control absoluto y en microsegundos de la física económica del simulador SaaS.
- Si hay un error legal publicado en el DOF, el Admin no tiene que esperar a que el equipo de soporte del gobierno repare su base de datos.
- Preparamos el camino fundamental para futuros ajustes globales (Tasas Libor, Intereses, Modulaciones de Layout, Gestión de Tier de la organización).

## Mitigaciones
- Para evitar que un Admin destruya el sistema al tipear por accidente `$1,150.00` en lugar de `$115.00` UMA, el componente de escritura debe heredar los blindajes matemáticos de "Fat Finger" del ADR-022, bloqueando iteraciones que desvíen más del 15% del ancla legal anualizada y emitiendo un "Adversarial Veto".
