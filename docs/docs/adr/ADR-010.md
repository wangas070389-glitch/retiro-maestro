# ADR-010: Server-First Architecture & Sovereign Persistence

## Status
Accepted

## Context
Retiro Maestro currently operates as a client-side SPA (Single Page Applicaton). While fast, it lacks:
1.  **Persistence**: Data is lost on refresh/device switch.
2.  **Security**: Calculation logic is exposed to the client.
3.  **Sovereignty**: Users cannot "own" their long-term simulation history.

## Decision
We will migrate the application to a **Server-First Architecture** using:
*   **Meta-Framework**: Next.js 14+ (App Router)
*   **Database**: PostgreSQL 16
*   **ORM**: Prisma (for Type Safety)
*   **Deployment**: Docker Containers (Portability)

## Consequences

### Positive
*   **Truth Source**: The Server becomes the authoritative source of "Net Pension" calculations.
*   **Persistence**: User simulations are stored durably.
*   **Security**: Sensitive logic (e.g., future tax engines) can be hidden from the browser.

### Negative
*   **Complexity**: Requires managing a backend, migrations, and API routes.
*   **Cost**: Database hosting is not free (unlike static SPA hosting).
*   **Latency**: Server interactions introduce network delays.

## Compliance
*   **Hybrid Compute**: To mitigate latency (see RED-010), the `PensionEngine` will exist in two forms: a lightweight Client version for sliders, and a heavy Server version for final verification.
*   **Sovereignty**: The database schema must be exportable (JSON/CSV) to ensure users can retreat to a "Sovereign" local file if needed.
