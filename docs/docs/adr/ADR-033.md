# Decoupled Profile Server Actions (ADR-033)

## Context
Standard users require the ability to update their personal information (Name, Email) and their passwords. We must decide how to structure the API layer handling these mutations. The codebase is currently rapidly transitioning to React Server Actions (`'use server'`) over traditional REST API route handlers to maximize typing parity and reduce bridging logic.

## Decision
We will execute all Sovereign Profile mutations exclusively through **Decoupled Server Actions** hosted within a dedicated `src/actions/user-actions.ts` file, completely segregating the identity flow from the administrative flow (`admin-actions.ts`). Furthermore, we will bisect the actions into two distinct functions: `updateProfileInfoAction` and `updatePasswordAction`.

## Consequences

### Positive
* **Isolation of Concerns:** The general profile update action is not bottlenecked or unnecessarily complicated by the heavy `bcrypt` hashing mechanics required for the password update.
* **Granular Security:** We can apply specific, rigid checks to `updatePasswordAction` (e.g., verifying the old password hash) without slowing down the lighter `updateProfileInfoAction`.
* **UI Independence:** The Client Component can mount two separate forms with two separate submit handlers and independent loading states, preventing the UI from freezing entirely while waiting for a password hash calculation.

### Negative
* Requires the Client Component to manage duplicate error handling mechanisms and toast invocations for the two separate forms.
