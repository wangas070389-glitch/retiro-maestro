# ADR-037: Dynamic User Persona Classification and Delegated Advisor Operations

## Status
**ACTIVE**

## Context
With the rollout of the B2B2C Advisor CRM and interactive features like the "Plan de Aportaciones" (Agenda de Pagos), the system requires:
1. Contextual, automated diagnostic guidance for clients based on their age, employment status, weeks, and rights conservation state.
2. A way for advisors to impersonate client workspaces and perform critical mutations (e.g. saving simulations, activating strategies, toggling monthly payments) directly on the client's database record rather than their own.
3. Strict backwards compatibility when loading simulation records created before the introduction of new input parameters.

## Decision
1. **Deterministic Persona Classification**: Implement a 5-Group classification rule inside `PersonaClassifier.ts` to categorize users dynamically based on `age`, `is_ongoing_work`, `weeks`, and `last_termination_date`.
2. **Glassmorphic Diagnostics UI**: Show custom advisory alert blocks on the main dashboard for users and advisors containing clear checklists, warning indicators, and next steps in Spanish.
3. **Delegated Execution Tunnel**: Update all key dashboard actions (e.g. `saveSimulationAction`, `selectStrategyAction`, `togglePaymentStatusAction`) to accept an optional `clientId` parameter. When present, the server actions verify that the session user is an authorized advisor for that client, then query/update the client's DB record.
4. **Zod Backwards Compatibility Guards**: Refactor `PensionSchema` in `calculate-pension.ts` to include `.default()` fallbacks for the new family allowance fields (`has_wife`, `children_count`, and `dependent_parents_count`). This ensures that legacy simulation records that lack these properties will parse safely instead of triggering validation crashes.

## Consequences
- **Positive**: Backwards compatibility is fully preserved, preventing crashes on existing user accounts. Impersonated client states are successfully synchronized in real time.
- **Negative**: Adds validation logic overhead for all action operations. Require strict verification of `clientId` parameter usage to prevent security leaks.
