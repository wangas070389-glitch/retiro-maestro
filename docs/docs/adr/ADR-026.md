# ADR-026: Edge-Based Client-Side PDF Extraction

## Context and Problem Statement
To extract data from the *Constancia de Semanas Cotizadas*, we need a mechanism to read PDF binaries. Traditionally, this is sent to a server for processing via `pdftotext` or Python libraries. However, transferring sensitive PII (NSS, CURP, DOB, Salary) to the server contradicts our Data Sovereignty mandate if it can be avoided.

## Decision Drivers
- **Data Privacy**: Minimizing PII transit.
- **Latency**: Instantaneous feedback to the user upon dropping the file.
- **Server Load**: Offloading heavy binary parsing from the Node/Next.js backend.

## Considered Options
1. **Server-Side API Route**: Send `FormData` to a Next.js API route that uses `pdf-parse`.
2. **Client-Side WASM/JS**: Use `pdfjs-dist` entirely within the browser's context.

## Decision Outcome
**Chosen option: "Option 2 (Client-Side WASM/JS)"**, because it fulfills the Data Sovereignty requirement by parsing the document locally in the user's browser edge worker.

### Positive Consequences
- True zero-knowledge ingestion. We only persist the final sanitized state if they save the Dossier.
- Zero latency for uploading large PDFs; parsing speed depends solely on the local CPU.

### Negative Consequences
- `pdfjs-dist` adds to the client bundle size (handled via dynamic imports to mitigate initial load).
- Browser memory usage may spike momentarily during extraction.
