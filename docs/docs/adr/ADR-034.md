# Polymorphic User Model for Advisor Mapping (ADR-034)

## Context
With the shift to a B2B2C model, we need a way for professional Advisors to manage their clients. We must decide whether to create a separate `Advisor` database table and model, or maintain a single `User` table and use the `role` field to differentiate capabilities, utilizing a self-referential relation.

## Decision
We will use a **Single Polymorphic User Table** with a Self-Referential Foreign Key.
1.  We will add `ADVISOR` as a valid string to the `role` scalar parameter.
2.  We will add `advisorId` (String, Optional) to the `User` model, pointing to `User.id`.
3.  We will add a relation array `clients User[] @relation("AdvisorClients")` for reverse lookups.

## Consequences

### Positive
*   **Unified Authentication:** NextAuth handles one single `User` table. We do not need complex multi-tenant or multi-table JWT callbacks.
*   **Fluid Transitions:** A standard `USER` can be upgraded to an `ADVISOR` via the Super Tablero simply by changing a string field, rather than migrating data between tables.
*   **Simplified Schema:** Queries for Dossiers and Simulations remain attached to a single universal `userId`.

### Negative
*   **Careful Filtering Required:** Queries asking for `db.user.findMany()` must now explicitly implement `where: { role: 'USER' }` if they want to exclude advisors from standard lists.
*   **Self-Referential Complexity:** Care must be taken so an Advisor cannot be assigned to another Advisor, or to themselves, requiring software-level validation loops in the assignment actions.
